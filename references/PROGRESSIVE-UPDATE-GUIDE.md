## 🎯 渐进式更新策略

### 为什么选择渐进式？

1. **风险控制** - 避免一次性切换导致的大面积故障
2. **可回滚** - 每个阶段都可以快速回退
3. **数据驱动** - 基于实际数据决策是否进入下一阶段
4. **用户无感知** - 更新过程对用户透明

**进入下一阶段的条件**:
1. 当前阶段所有验收标准达标

3. 用户反馈无负面

**回滚条件**:
1. 准确率低于告警阈值
2. 错误率超过 5%
3. 收到用户投诉

**回滚操作**:
```bash

# 修改 intent-config.json 中的 phase 字段

# 重启服务（如果需要）
systemctl restart salesmartly-copilot
```

## 🚨 故障处理

### 常见问题

#### 1. 意图识别准确率突然下降

**可能原因**:
- LLM 模型不稳定
- 新增触发词冲突
- 用户输入模式变化

**处理步骤**:
1. 检查日志，确认问题范围
2. 回滚到上一阶段配置
3. 分析错误案例
4. 修复后重新测试

#### 2. 响应时间过长

**可能原因**:
- LLM 调用超时
- 网络问题
- 并发量过大

**处理步骤**:
1. 检查 LLM API 状态
2. 增加超时时间或重试次数
3. 考虑限流或降级

#### 3. 用户投诉

**处理步骤**:
1. 立即回滚到稳定版本
2. 收集用户反馈
3. 分析问题原因
4. 修复后重新测试

## 🎓 经验总结

### 关键学习

1. **关键词兜底很重要** - 零成本，快速匹配 ~30% 请求
2. **模型自适应是核心** - 不同模型使用不同 Prompt 复杂度
3. **渐进式更新降低风险** - 避免一次性切换的故障
4. **监控指标驱动决策** - 数据说话，不凭感觉

### 最佳实践

- ✅ 每个阶段都有明确的验收标准
- ✅ 配置与代码分离，便于快速回滚
- ✅ 详细的日志记录，便于问题排查
- ✅ 用户反馈渠道畅通
