# SKILL.md 优化建议

**目标**: 提升 salesmartly-api-skills 在不同 LLM 模型下的意图识别准确率

**当前问题**: 不同客户使用不同模型，意图识别准确率差异大（60%-95%）

**创建时间**: 2026-03-24

---

## 📊 现状分析

### 模型能力差异

| 模型 | 意图识别准确率 | 主要问题 |
|------|--------------|---------|
| Qwen3.5-plus | 95%+ | 表现优秀 |
| Qwen3.5-flash | 85% | 复杂查询易错 |
| Qwen2.5-72B | 80% | 参数提取不准 |
| GPT-3.5-Turbo | 75% | 易混淆相似技能 |
| 小模型 (<7B) | 60% | 经常误解意图 |

### 根本原因

1. **SKILL.md 信息密度高** - 小模型难以快速定位关键信息
2. **缺少明确的触发词** - 弱模型无法快速匹配技能包
3. **示例不够结构化** - 弱模型难以模仿
4. **缺少降级策略** - 弱模型不知道何时澄清

---

### 方案 3: Few-shot 示例增强（优先级：中）

**位置**: 意图映射表之后

```markdown
## 📚 示例（AI 参考）

### 示例 1: 简单查询（所有模型必须掌握）

**用户**: "昨天的客户有哪些？"

**AI 思考过程**:
```
1. 关键词识别："昨天" + "客户" → 查询客户
2. 时间提取："昨天" → days=1
3. 意图判断：新增客户 → filter-by=created
4. 脚本选择：query-customers.py
5. 置信度评估：0.95（非常明确）
```

**调用命令**:
```bash
uv run scripts/query-customers.py --days 1 --filter-by created
```

**预期输出**:
```
✅ 昨天新增 15 个客户：
1. 张三 - 13800138000
2. 李四 - 13900139000
...
```

---

### 示例 2: 复杂查询（L2+ 模型掌握）

**用户**: "最近 7 天 VIP 客户的聊天记录"

**AI 思考过程**:
```
1. 关键词识别："最近 7 天" + "VIP 客户" + "聊天记录"
2. 时间提取："最近 7 天" → days=7
3. 筛选条件："VIP" → tags="VIP"
4. 意图判断：查询聊天记录 → query-messages.py
5. 置信度评估：0.85（明确，但需要多步推理）
```

**调用命令**:
```bash
# 步骤 1: 先查询 VIP 客户
uv run scripts/query-customers.py --days 7 --tags "VIP"

# 步骤 2: 再查询聊天记录（需要客户 ID）
uv run scripts/query-messages.py --chat-user-id {客户 ID} --days 7
```

**注意**: 这是多步骤查询，需要先获取客户 ID

### 方案 4: 降级策略明确化（优先级：中）

**位置**: 示例之后

```markdown
## ⚠️ 降级策略（AI 必读！）

### 何时必须澄清？

**出现以下情况时，不要调用脚本，先问清楚：**

| 情况 | 示例 | 正确做法 | 错误做法 |
|------|------|---------|---------|
| 时间不明确 | "查客户" | 问："请问查哪个时间范围？" | ❌ 直接调用（可能查错） |
| 对象不明确 | "分配客户" | 问："请问分配哪个客户？给哪个客服？" | ❌ 直接调用（参数缺失） |
| 动作不明确 | "客户张三" | 问："请问要查询、更新还是分配？" | ❌ 直接调用（可能做错） |
| 参数冲突 | "昨天和今天的客户" | 问："请问查昨天还是今天？" | ❌ 直接调用（逻辑冲突） |
| 多步骤操作 | "给所有 VIP 客户发消息" | 先确认："找到 156 个 VIP 客户，确认要群发吗？" | ❌ 直接群发（可能骚扰） |

### 澄清话术模板

```
我没理解{不明确的部分}，请问您是想：
1. {选项 1}
2. {选项 2}
3. {选项 3}

或者您可以更详细地描述您的需求。
```

### 置信度自评

**调用脚本前，自我评估置信度：**

```
置信度≥0.8  → ✅ 直接调用
置信度 0.5-0.8 → ⚠️ 可以调用，但告知用户假设
置信度<0.5  → ❌ 必须澄清，不要调用
```

**示例**:
```
⚠️ 我理解您想查询昨天的客户，对吗？
（如果不对，请说明具体时间范围）

✅ 确认无误，正在查询...
```
```

### 方案 5: 参数验证规则（优先级：低）

```markdown
## ✅ 参数验证（AI 必须检查！）

### 必填参数检查

| 脚本 | 必填参数 | 缺失时处理 |
|------|---------|-----------|
| `assign-session.py` | `--chat-user-id`, `--member-id` | 问："请问分配给哪个客服？" |
| `query-messages.py` | `--chat-user-id` 或 `--session-id` | 问："请问查哪个客户的聊天记录？" |
| `update-customer.py` | `--chat-user-id` | 问："请问更新哪个客户？" |

### 参数格式验证

| 参数 | 正确格式 | 错误格式 | 验证规则 |
|------|---------|---------|---------|
| `--days` | `7` (整数) | "七天" | 必须是数字 |
| `--date` | `2026-03-24` | "昨天" | YYYY-MM-DD 格式 |
| `--phone` | `8613800138000` | `13800138000` | 加区号，不含 + |
| `--chat-user-ids` | `id1,id2,id3` | `["id1","id2"]` | 逗号分隔字符串 |

### 参数边界检查

| 参数 | 最小值 | 最大值 | 超出处理 |
|------|-------|-------|---------|
| `--page-size` | 1 | 100 | 自动修正为 100 |
| `--days` | 1 | 365 | 问："时间范围太大，确定吗？" |
| `--chat-user-ids` | 1 个 | 100 个 | 超过 100 个时分批处理
```
